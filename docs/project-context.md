# Project Codex Context

## 1. Project Overview
- **Description:** Repaso App is an educational platform for studying and reviewing professional exams through exam-style questions, progress tracking, and educational content. It was born focused on the Psychology Board Exam (Puerto Rico) and is designed as a **whitelabel** architecture to support multiple exams and brands.
- **Target users:** Students preparing for professional exams; in the future, instructors and administrators.
- **Value proposition:** Structured, reusable, and scalable study experience, with measurable progress, localized content, and a technical foundation ready for multiple educational verticals.

## 2. Platforms & Tech Stack

### Frontend (Web)
- **Framework:** Next.js 16 App Router
- **Language:** TypeScript
- **UI / Styling:** Tailwind CSS v4
- **State:** React Hooks via shared `packages/hooks`
- **Features:** route groups, role-based dashboard switcher, student MVP flows

### Mobile
- **Platform:** iOS / Android
- **Framework:** Expo (React Native)
- **State:** Shared via `packages/hooks`
- **Project status:** Future scaffold (no features implemented)

### Backend
- **Platform:** Supabase
- **Runtime:** Serverless (Edge Functions)
- **API:** Supabase client (PostgREST + RPC + Edge Functions)
- **Auth:** Supabase Auth

### Database
- **Engine:** PostgreSQL (Supabase)
- **Schema:** Versioned SQL migrations
- **Migrations:** Supabase migrations (`infra/database/supabase/migrations`)
- **Seeds:** SQL seeds (`infra/database/supabase/seeds`)

### Infrastructure
- **Monorepo:** npm workspaces
- **CI/CD:** GitHub Actions
  - Web deploy
  - Mobile build
  - Supabase migrations
- **Environments:** Local (Supabase local), Cloud (Supabase hosted)

## 3. System Architecture
- **Components:**
  - Web App (Next.js)
  - Mobile App (Expo)
  - Supabase (Auth, DB, Storage, Edge Functions)
  - Shared packages (`domain`, `application`, `infrastructure`, `hooks`)
- **Interaction:**
  - Frontends → Supabase (auth + data)
  - Supabase → Postgres / Storage / Edge Functions
- **Boundaries:**
  - Frontend: UI + role shells + shared React hooks
  - Application: namespaced use cases and neutral access orchestration
  - Infrastructure: namespaced Supabase auth, persistence, and repository implementations
  - Backend: critical business rules in DB/RLS and Edge Functions
  - DB: source of truth
- **Shared code:** `packages/domain`, `packages/application`, `packages/infrastructure`, `packages/hooks`
- **Web-specific dependency assembly:** `apps/web/src/lib/repaso-dependencies.ts`

### Shared Package API Shape
- Package roots now expose **namespaced exports only**:
  - `@repaso/domain`: `Access`, `Admin`, `Instructor`, `Owner`, `Shared`, `Student`
  - `@repaso/application`: `Access`, `Admin`, `Instructor`, `Owner`, `Student`
  - `@repaso/infrastructure`: `Access`, `Admin`, `Instructor`, `Owner`, `Shared`, `Student`
  - `@repaso/hooks`: `Access`, `Admin`, `Instructor`, `Owner`, `Student`
- Access orchestration is centralized in `packages/application/src/access`.
- Access policy and role predicates are centralized in `packages/domain/src/access`.
- Role-aware business logic must not live under `shared`; `shared` is reserved for primitive/base utilities and types.

### Current Access Composition
- `CurrentAccess` is resolved through the neutral package stack:
  - `@repaso/infrastructure.Access.createSupabaseAccessRepository`
  - `@repaso/application.Access.resolveCurrentAccess`
  - `@repaso/hooks.Access.CurrentAccessProvider` / `@repaso/hooks.Access.useCurrentAccess`
- `CurrentAccess` includes first-class booleans for:
  - `isStudent`
  - `isInstructor`
  - `isAdmin`
  - `isOwner`
- Web role authorization should delegate to `@repaso/domain.Access` instead of duplicating local role comparisons.

### Current Web Routing Shape
- Student routes remain user-facing at:
  - `/dashboard`
  - `/topics`
  - `/topics/[slug]`
  - `/topics/[slug]/practice`
  - `/topics/[slug]/practice/summary`
  - `/search`
  - `/attempts/[attemptId]`
- Internal route groups separate:
  - `(student)`
  - `(owner)`
  - `(admin)`
  - `(instructor)`
- Non-student placeholder role routes currently exist at:
  - `/owner`
  - `/admin`
  - `/instructor`

## 4. API Contracts
- **Type:** Supabase autogenerated APIs + Edge Functions
- **Auth:** Supabase Auth (JWT)
- **Errors:** Supabase conventions (HTTP + error object)
- **Note:** No custom REST contracts defined yet

## 5. Data & Persistence Rules
- **Conventions:** snake_case in DB, camelCase in TS
- **Required fields:** defined in SQL migrations
- **Soft delete:** not defined
- **Auditing:** expected timestamps
- **Transactions:** managed by Supabase/Postgres
- **Validation:** DB (constraints) + client

## 6. Security & Auth Agreements
- **Auth:** Supabase Auth
- **Roles:** owner, admin, instructor, student
- **Authorization:** route/layout guards in web plus RLS-backed data constraints
- **Client policy source:** `@repaso/domain.Access` is the shared source of truth for app-level route authorization
- **Tokens:** JWT managed by Supabase
- **Critical rule:** never expose SERVICE_ROLE_KEY in client

## 8. Non-Goals & Constraints
- **Out of current scope:**
  - Payments (Stripe/PayPal only "ready")
  - Complete mobile features
- **Constraints:**
  - Psychology content in Spanish
  - Strong dependency on Supabase

## 9. Assumptions Codex Must Respect
- **Whitelabel** architecture is mandatory
- Supabase is the only backend (do not replace)
- Monorepo with shared packages
- SQL migrations as the source of truth for the schema
- Do not move sensitive logic to the client
- Do not simplify by removing RLS, roles, or package separation
- Web routes and components should consume shared hooks/use-cases and should not import `@repaso/infrastructure` directly; that wiring belongs in the web `lib` boundary
- Web code should consume package public APIs by namespace (for example `Domain.Access`, `Application.Access`, `Hooks.Access`) rather than package-internal source paths
